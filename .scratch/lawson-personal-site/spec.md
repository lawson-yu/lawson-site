# LAWSON 个人技术站点

Status: ready-for-agent

## Problem Statement

LAWSON 需要一个可长期运营的个人技术站点，向正在学习 AI 与进行工程实践的人展示真实项目、经验、教学内容与精选 GitHub 项目，同时为技术搜索与面试提供可信入口。现有应用只有公开首页占位，无法发布内容、管理项目、搜索内容或让本地 agent 安全导入定时生成的 Markdown。

## Solution

构建中文首发、以 `LAWSON — AI 与工程实践` 为定位的个人技术站点。公开站提供首页、博客、个人项目、精选项目、关于和搜索；作者后台只允许唯一作者身份通过 GitHub OAuth 登录，管理草稿与已发布内容。

本地 agent 通过受限导入 API 自动上传 Markdown 导入包及其本地图片，服务端保存图片并创建或更新草稿。所有内容从首发起按 locale 存储，为下一版英文内容与语言切换预留。站点采用深色、活泼、内容优先的原创技术编辑视觉，并满足 SEO、GEO、响应式与可访问性要求。

## User Stories

1. 作为正在学习 AI 的访客，我想看到 LAWSON 的明确定位和最新内容，以便判断站点是否值得关注。
2. 作为工程实践学习者，我想浏览原创实践文章，以便理解真实项目中的问题、决策和结果。
3. 作为读者，我想按博客内容类型浏览实践、教程和 AI 新闻，以便快速找到所需内容。
4. 作为读者，我想按主题标签筛选内容，以便围绕 AI、工程或其他主题持续学习。
5. 作为读者，我想搜索已发布内容，以便通过关键词找到相关文章、项目或精选项目。
6. 作为读者，我想在搜索结果中按内容类型筛选，以便缩小结果范围。
7. 作为读者，我想阅读结构清晰、代码可复制的技术文章，以便实际应用所学内容。
8. 作为读者，我想看到文章作者、发布日期、更新时间和外部来源，以便判断内容的可信度与时效性。
9. 作为搜索引擎访客，我想每篇公开内容拥有稳定 URL、准确元数据和站点地图，以便发现与索引站点内容。
10. 作为生成式搜索产品的使用者，我想读取回答具体问题、引用来源并标注作者的内容，以便更可靠地理解和引用其结论。
11. 作为招聘方，我想浏览 LAWSON 的个人项目、技术栈、成果、代码仓库和演示链接，以便评估其实践能力。
12. 作为读者，我想浏览独立的精选项目资料库，以便理解值得关注的 GitHub 项目解决什么问题、适用于什么场景。
13. 作为读者，我想按周和主题浏览精选项目，以便追踪值得关注的工程与 AI 工具。
14. 作为访客，我想通过 GitHub、LinkedIn 和邮箱联系 LAWSON，以便进一步交流。
15. 作为移动端访客，我想在窄屏舒适阅读、导航和搜索，以便不依赖桌面设备。
16. 作为偏好减少动态效果的访客，我想关闭进入和漂浮动画，以便舒适使用站点。
17. 作为作者，我想通过 GitHub OAuth 进入作者后台，以便不维护密码和注册流程。
18. 作为作者，我想创建、编辑、预览、发布和撤回博客、个人项目与精选项目，以便维护公开内容。
19. 作为作者，我想让新标签先处于待确认状态，以便避免同义或拼写不同的标签污染筛选体系。
20. 作为作者，我想让后台只显示自己的受保护内容，以便保证单作者管理边界。
21. 作为运行本地 agent 的作者，我想自动提交 Markdown 和本地图片，以便将定时生成的候选内容快速变成后台草稿。
22. 作为运行本地 agent 的作者，我想使用单独可轮换的导入密钥，以便不暴露 Supabase 管理凭证。
23. 作为运行本地 agent 的作者，我想重复导入同一 `externalId` 时更新同一草稿，以便定时任务重跑不会产生重复内容。
24. 作为作者，我想让 agent 对已发布内容的更新变成待审核更新草稿，以便自动化不会意外修改公开内容。
25. 作为作者，我想让 Markdown 中的本地相对路径图片被安全保存为受管图片，以便公开内容不依赖本机路径或不稳定外链。
26. 作为未来英文读者，我想让内容具有 locale，以便下一版可增加英文翻译而不迁移既有数据。
27. 作为维护者，我想让新增页面和组件遵守已记录的视觉规范，以便站点在新会话和长期迭代中保持一致。
28. 作为维护者，我想通过浏览器和自动化检查验证公开流程、作者流程和导入流程，以便持续修改不会破坏核心行为。

## Implementation Decisions

- 站点为单体 Next.js App Router 应用，部署在 Vercel；托管 Supabase 负责认证、关系数据、全文搜索与图片存储。
- 公开信息架构包含：首页、博客列表与详情、个人项目、精选项目、关于、搜索。作者后台与登录页不出现在公开导航。
- `LAWSON` 是公开作者和个人品牌名称。首页定位为“LAWSON — AI 与工程实践”，并说明真实项目、AI 工具与工程问题拆解、GitHub 项目精选。
- 视觉规范以现有 `DESIGN.md` 为单一事实来源。采用近黑阅读面、深海军蓝分区、灰白文字、品牌蓝、洋红栏目强调与亮黄主操作；使用原创抽象 AI/工程插画和波浪/山形背景，绝不复制第三方人物、Logo、插画、文案或页面结构。
- 页面动效只包括克制的首屏依序淡入上移与低幅背景漂浮；必须遵从 `prefers-reduced-motion`，禁止自动播放声音、视频和高频闪烁。
- 所有内容使用共同核心字段：`kind`、`title`、`summary`、`locale`、`tags`、正文、状态、创建和更新时间。按内容类型增加项目或精选项目字段；未知字段作为扩展元数据保留。
- 内容类型固定为博客、个人项目、精选项目；博客再以实践、教程、AI 新闻分类。AI 新闻属于博客，不是独立内容系统。
- 个人项目包含问题、成果、技术栈、仓库、演示、封面、置顶和公开状态。精选项目包含仓库来源、解决问题、用途、技术标签、作者短评、收录日期和按周聚合能力。
- 公开内容只有草稿和已发布两态。发布即时生效；首发不提供定时发布、版本历史或多人审核。
- GitHub OAuth 认证后，服务端以稳定 GitHub 用户 ID 判断唯一作者身份；绝不能仅以“已登录”、用户名或用户可编辑 metadata 授权。
- 公开访问只读取已发布内容；作者后台与写入操作仅限唯一作者。所有暴露给数据 API 的表启用 RLS，并按实际访问模型制定策略。
- 自动导入为受限 HTTP 接口。调用方使用独立、可轮换的长随机导入密钥；接口仅可创建或更新草稿，不可发布、读取私有内容、持有 Supabase `service_role` 或越过数据库授权边界。
- 自动导入由受信任的服务端执行器在校验导入密钥后处理。agent 永不获得 `service_role` 或数据库连接；为安全写入受管图片，执行器可在服务端持有 `service_role`，且只可执行受限导入流程，不能读取私有内容或发布。详情见 `technical-design.md` 与 ADR 0001。
- 导入包通过 `multipart/form-data` 上传一份 Markdown 及其相对路径图片。服务端校验密钥、文件类型、大小、相对路径和 Markdown front matter，上传图片到受管存储，再将正文引用改写为受管 URL。
- 导入文档强制包含 `externalId`。相同 ID 更新既有草稿；若目标为已发布内容，则生成待审核更新草稿，不覆盖公开内容。
- Markdown 支持标题、列表、链接、引用、表格、代码块和图片；禁止原始 HTML 和脚本。代码块支持语法高亮和复制。
- 内容从首发起保存显式 locale；首发默认 `zh-CN`，不提供语言切换或强制翻译。后续英文内容以同一内容关系的 locale 版本加入。
- 搜索使用 Supabase Postgres 全文搜索，索引已发布内容的标题、摘要、正文和标签；搜索页支持关键词与内容类型筛选，且不参与外部搜索引擎索引。
- 标签由受控标签库管理。导入只能直接使用已确认标签；新标签作为待确认项创建，需作者确认后才在公开筛选中使用。
- SEO 基线包括稳定 URL、标题和摘要元数据、Open Graph 图、sitemap、RSS、canonical URL。GEO 基线包括语义化标题、明确问题回答、作者与日期、外部事实来源、`Article`、`Person` 和 `Project` 结构化数据。
- 首发公开联系入口只含 GitHub、LinkedIn、邮箱和 RSS；不实现邮件订阅、站内消息、评论或用户账户。
- 首发使用设计规范定义的 `Geist` 无衬线字体、间距、圆角和边框 token；新页面与组件不得自行定义未记录的颜色、字号、圆角或动效变体。
- 浏览器与 Client Component 通过 Next.js Route Handler（`/api/*`）读取和写入后端数据；Server Component、metadata、sitemap 与 RSS 直接调用同一领域模块，不请求本站自己的 Route Handler，以保留构建期预渲染并避免内部 HTTP 往返。
- Route Handler 负责 HTTP、认证与输入输出映射；可使用 Zod 校验所有不可信输入，业务规则仍封装在领域模块中。
- 页面过长时优先拆为该路由私有的页面级组件；只有已经在多个页面稳定复用的视觉或交互模式才提升为全局组件。
- 所有页面与组件样式统一使用 Tailwind CSS，并遵循 `DESIGN.md` 的 token、间距、断点和动效规则；不新增 CSS Modules、CSS-in-JS 或页面内联样式。`app/globals.css` 只承载 Tailwind 入口、设计 token 与必要的全局基础规则。

## Testing Decisions

- 测试验证用户可观察到的行为、权限和数据边界，不断言内部组件层级、CSS class 名称或私有实现细节。
- 公开站以真实浏览器 e2e 为最高 seam：从 App Router 路由验证首页、内容列表、详情、搜索、键盘导航、响应式布局和减少动态效果偏好。
- 使用现实但稳定的已发布内容夹具，验证首页内容入口、博客与项目详情、标签筛选、搜索结果和不索引搜索页的元数据。
- 作者后台以浏览器 e2e 验证：未登录访问受保护区域被拦截；唯一作者身份可创建草稿、编辑、发布和管理待确认标签；其他身份不能写入。
- 自动导入以 HTTP 路由契约测试验证：缺失或错误密钥被拒绝、有效导入包创建草稿、相对图片被存储并重写、重复 `externalId` 幂等、已发布内容不会被 agent 覆盖、无效文件与原始 HTML 被拒绝或安全处理。
- Supabase 迁移和 RLS 以数据库级验证确认：匿名访问只能读取已发布内容，作者权限限定写入，导入边界不能获得服务角色权限。
- 延续现有公开首页 Playwright smoke test，并按新增用户可见路径扩展 e2e；静态门槛为 lint、类型检查、生产构建和 e2e。
- 每个公开页面在 `320px`、`768px`、`1024px`、`1440px` 检查；视觉验收遵从 `DESIGN.md`，并检查无横向滚动、清晰焦点、有效 alt 文本与可读对比度。
- 实现遵循已确认的 `technical-design.md`：公开 canonical URL 使用 `/{locale}/...`（首发 `zh-CN`，无 locale 入口重定向）；内容以 identity 下的当前草稿与当前已发布 variant 建模，不保留版本历史。
- Route Handler 契约测试覆盖 Zod 校验、认证、状态码与 DTO；领域模块测试覆盖业务规则，Server Component 与 metadata 测试不依赖对本站 API 的 HTTP mock。

## Out of Scope

- 邮件订阅、评论、站内消息、公开注册、读者账户、多人协作、投稿流程。
- 定时发布、版本历史、多人审核、agent 自动发布。
- 第三方搜索服务、客户端全量下载搜索、复杂推荐算法。
- 首发英文翻译与语言切换器；仅实现多语言数据预留。
- 复制任何参考站的受版权保护视觉资产、人物、插画、文案或页面结构。
- 自建常驻服务器、agent 直接连接数据库、agent 持有 Supabase 管理密钥。

## Further Notes

- `CONTEXT.md` 定义本功能的规范术语；实现和新增命名应采用其中的词汇。
- `docs/adr/0001-restricted-agent-import.md` 记录自动导入安全边界；`docs/adr/0002-locale-aware-content-from-launch.md` 记录 locale-first 数据决策。
- `DESIGN.md` 是前端设计规范。它在实现第一个公开视觉 ticket 前可继续细化，但不能绕开本规格中的范围和可访问性约束。
- 生产配置仍需作者提供 GitHub OAuth 应用配置、唯一 GitHub 用户 ID、Supabase 项目变量、导入密钥、公开联系链接和域名。
