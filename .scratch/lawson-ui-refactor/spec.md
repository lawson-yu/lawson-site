# Lawson Site 全页面 UI 重构

Status: ready-for-agent

## Problem Statement

作为 LAWSON 站点的维护者，我希望在不改变站点现有内容、路由、数据、权限或操作流程的前提下，依据当前 `DESIGN.md` 重构所有既有页面的前端 UI。现有公开站点、登录页与作者后台使用的深色技术编辑风、组件形态和响应式行为，与新的浅冰蓝画布、轻量排版、细边界及深色反向区块的设计规范不一致；移动端公开导航还会仅靠自动换行而拥挤。

## Solution

将新的 Design System 应用于所有现有页面：公开内容页根据 Home、Catalogue 与 Article Detail 的 surface recipe 重构；登录页和作者后台根据 Todo workspace 的浅色工作区语法重构。保留 LAWSON、博客、项目展示、精选项目、搜索、联系入口、作者后台及其已有内容与行为；不得伪装成课程站或添加不存在的产品内容。

使用系统字体栈代替规范中的缺失本地字体，并记录该 Design Spec Gap。公开页在紧凑与移动宽度使用可访问的 hamburger 菜单。移除当前首页入场动画，不用其他动效替代。以已有公开体验和作者生命周期的浏览器测试作为最高验收缝隙，并在真实作者登录会话中完成桌面与移动视觉巡检。

## User Stories

1. 作为首次访问者，我想在浅冰蓝画布上看到清晰的 LAWSON 个人技术品牌内容，以便立即理解这是 AI 与工程实践站点。
2. 作为公开站点读者，我想在首页继续看到现有的最新内容、个人项目、精选项目与联系方式，以便视觉更新不改变站点的信息价值。
3. 作为博客读者，我想在博客列表以高密度、可扫描的 catalogue 布局浏览真实已发布文章，以便快速发现内容。
4. 作为项目访客，我想在项目列表浏览真实项目摘要与已有封面，以便判断哪些实践值得继续阅读。
5. 作为精选项目读者，我想在精选项目列表继续使用现有筛选与搜索能力，以便按主题发现作者的精选解读。
6. 作为文章读者，我想在博客详情得到低干扰、可长时间阅读的文章排版，以便阅读正文、代码和图片时保持专注。
7. 作为项目详情读者，我想在不丢失问题、产出、技术栈和外部链接的情况下阅读项目详情，以便理解作者的实践成果。
8. 作为精选项目详情读者，我想继续看到作者的判断、来源与内容正文，以便区别精选解读与普通转载。
9. 作为搜索访客，我想继续输入关键词并筛选内容类型，以便找到已发布内容。
10. 作为关于页访客，我想在统一的视觉语言中理解作者与站点定位，以便决定是否继续浏览或联系。
11. 作为联系站点作者的访客，我想继续找到 GitHub、LinkedIn、邮箱和 RSS 链接，以便使用已有的联系入口。
12. 作为小屏访客，我想用可访问的菜单开关浏览全部公开导航，以便链接不会因自动换行而难以使用。
13. 作为键盘访客，我想在 logo、菜单、链接、搜索和操作控件上看到清晰焦点，以便不使用鼠标也能操作站点。
14. 作为使用窄屏设备的访客，我想让页面按设计断点堆叠且不产生横向滚动，以便稳定阅读。
15. 作为内容读者，我想在有真实封面时看到比例正确的媒体，在无封面时仍得到清晰的文本卡片，以便不被虚构图片误导。
16. 作为登录访客，我想在登录页获得与公开站点一致但简洁的认证入口，以便明确下一步操作。
17. 作为唯一作者身份，我想在作者后台查看博客、项目、精选项目和标签工作区，以便继续管理既有内容。
18. 作为作者，我想在新建与编辑表单中清晰区分输入、状态、生命周期操作和媒体管理，以便降低内容维护负担。
19. 作为作者，我想在博客、项目和精选项目预览页阅读接近公开详情的内容样式，以便在发布前判断呈现效果。
20. 作为作者，我想继续上传、删除和引用受管媒体，以便视觉重构不破坏既有媒体工作流。
21. 作为作者，我想继续执行草稿、发布、取消发布和标签确认等已有操作，以便 UI 变更不改变内容状态机。
22. 作为偏好减少动态效果的访客，我想得到稳定的静态页面，以便不会被非必要动画干扰。
23. 作为站点维护者，我想用重复的颜色、间距、边界、圆角和排版 token 管理 UI，以便后续页面保持一致。
24. 作为站点维护者，我想保留现有的 locale、公开 URL、SEO、RSS 和 sitemap 行为，以便视觉重构不影响内容发现。
25. 作为验收者，我想同时在桌面和移动视口查看公开页与真实作者工作区，以便确认全页面 UI 已被覆盖。

## Implementation Decisions

- 所有既有页面均在范围内：公开首页、关于、博客、项目展示、精选项目、搜索、各内容详情、登录、作者工作区、作者的新建/编辑页与预览页。服务端 API、数据 schema、RLS、导入契约、发布状态机、内容查询、路由和 locale 行为不变。
- 公开页采用新的 Design System token：浅冰蓝 canvas、深蓝黑 inverse surface、淡蓝 soft surface、蓝色 primary action、海军蓝 selected state、细边界、无阴影卡片和 6px/媒体圆角。不得把浅色页面改为白底或把目录卡片改为白底阴影卡片。
- 公开首页使用 Home recipe；博客、项目和精选项目列表按 Catalogue 的高密度列表/卡片语言适配；博客、项目和精选项目详情按 Article Detail 的阅读语言适配；搜索和关于页采用同一浅色公开 shell 的相应布局，而不是引入课程站的新信息架构。
- 作者后台、表单和预览采用 Todo workspace 的浅色 hero、双栏工作区、可见状态与低密度规则侧栏语法。其目标是内容管理，不引入任务排序、日期、持久化机制或 Todo 业务实体。
- 现有内容优先于参考站内容。不得新增课程、注册、newsletter、社区、虚构 CTA、课程筛选、课程分页、课程图片或额外 footer 链接；保留现有公开导航、内容数据、作者操作、联系链接和真实外部链接。
- 媒体仅使用既有真实资源与内容提供的真实封面 URL。有封面时采用设计规范的响应式媒体比例和圆角；无封面时维持文本卡片，不生成占位图、不复用 River Hero、也不要求补充内容数据。
- 规范列出的本地 Aeonik Mono 与 TWK Lausanne 字体文件不在仓库中。本次使用系统 monospace/sans-serif fallback 来分别承担 UI/metadata 与 display/body 的视觉角色；此为已确认的 Design Spec Gap，不假称本地字体已加载。
- 当前项目继续使用既有 Next.js 与 Tailwind CSS 构建方式；将设计 token 和组件契约实现为该栈中的全局样式和页面样式，不为了参考实现迁移到原生 HTML/CSS，也不新增 Zustand 或其他状态管理。
- 公开导航在 `<=991px` 以可访问 hamburger 取代自动换行：控制按钮具有可见文字或可访问名称、`aria-expanded` 与 `aria-controls`，键盘可操作，展开后可到达所有已有导航链接。此交互仅服务响应式导航，不改变导航目的地。
- 响应式断点为 mobile `<=767px`、compact `768px–991px`、desktop `>=992px`。桌面与移动的容器、网格、标题与工作区按规范收缩或堆叠；除详情相关的明确横向阅读 surface 外，页面不得产生横向滚动。
- 移除当前首页进入动画及相关非必要 transition/animation；不加入替代动效。保留或加强 `:focus-visible`，并继续尊重 `prefers-reduced-motion`。
- 保持语义 landmark、heading 层级、链接与按钮语义、已有 `aria-*` 关系、有意义的替代文字、搜索/筛选的可访问名称和 live feedback。视觉重构不得以 div 模拟交互控件。
- 本次以 `DESIGN.md` 的无动效浅色视觉契约为 UI 决策来源，优先于 `CONTEXT.md` 中尚未同步的深色河流、波纹和动效描述。该词汇表冲突需在实现交付时明确记录，不能静默混合两套视觉语言。

## Testing Decisions

- 以访客和作者可感知的外部行为为测试对象，不测试 Tailwind class、CSS 变量名称、组件内部状态或具体 DOM 实现。
- 优先使用现有公开体验 e2e 作为公开站点的最高测试缝隙：验证首页可访问、公开导航、内容列表/详情、搜索、RSS、sitemap、locale 和 SEO 基线仍可使用。
- 优先使用现有作者生命周期 e2e 作为作者后台的最高测试缝隙：验证真实作者身份进入工作区，完成博客、项目、精选项目的创建/编辑、预览和生命周期操作；验证非作者仍无法进入受保护区域。
- 为移动菜单补充或扩展一个以用户行为为中心的浏览器测试：小屏打开菜单、读取其展开语义、进入已有链接，并确保键盘可操作；不测试内部菜单组件的实现细节。
- 为无横向滚动、可见焦点、真实封面/无封面内容和减少动态效果补充最少的回归断言；页面内容与操作结果应继续来自真实服务端数据与现有测试种子。
- 执行项目既有静态验证与构建检查。完成后在 `1440×900` 和 `390×844` 对公开页和真实登录后的作者页进行视觉巡检；截图仅作本地证据，不提交到 `evidence/`。
- 可沿用现有首页、公开体验、搜索、项目、精选项目、作者博客、作者项目、作者精选项目与非作者 e2e 作为 prior art，而不是新建绕过认证的数据替身。

## Out of Scope

- 修改 Supabase schema、RLS、Storage、导入 API、导入密钥、作者身份判定、发布状态机或任何已有业务数据。
- 修改公开路由、locale 策略、SEO metadata 契约、RSS、sitemap、内容查询或搜索筛选语义。
- 增加课程、注册、付费、newsletter、社区、课程目录、远程筛选、假数据、额外 footer 链接、虚构媒体或占位插画。
- 采购、下载或加入未提供且未确认授权的本地字体；本次不声称像素级复刻这些字体。
- 引入 Tailwind 以外的新 UI 框架、全局客户端状态库、模态系统、动画体系或新的后端能力。
- 将 Todo recipe 扩展成真实任务管理功能，或以 UI 重构名义修改后台业务工作流。
- 提交本地环境变量、密钥、登录会话、`evidence/` 截图或其他本地验证产物。

## Further Notes

- 本规格来自已完成的 `$grill-me` 决策，不需要补充访谈；测试缝隙已由用户确认可接受。
- `DESIGN.md` 当前为用户工作区中的未提交修改，应视为本任务输入；实施时不得覆盖、重置或夹带修改其无关内容。
- `CONTEXT.md` 中的“视觉语言”“首屏互动视觉”和“页面动效”仍描述既有深色河流方案，和本规格冲突。本规格授权本次 UI 重构按新浅色、无动效规范执行；后续应单独决定是否同步更新领域词汇表与旧的首屏规格，避免文档长期漂移。
